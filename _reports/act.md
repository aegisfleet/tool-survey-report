# **act (nektos/act) 調査レポート**

## **1. 基本情報**

* **ツール名**: act (nektos/act)
* **ツールの読み方**: アクト
* **開発元**: nektos (コミュニティ主導のOSS)
* **公式サイト**: [https://nektosact.com/](https://nektosact.com/)
* **関連リンク**:
  * GitHub: [https://github.com/nektos/act](https://github.com/nektos/act)
  * ドキュメント: [https://nektosact.com/introduction.html](https://nektosact.com/introduction.html)
* **カテゴリ**: 開発ツール / CI/CD
* **概要**: GitHub Actionsのワークフロー定義ファイル（`.github/workflows`）を読み込み、Dockerコンテナを使用してローカル環境で実行するコマンドラインツール。「Think globally, act locally」を掲げ、クラウドへのプッシュなしでCI/CDパイプラインのテストを可能にする。

## **2. 目的と主な利用シーン**

* **解決する課題**:
  * GitHub Actionsの動作確認のために、修正のたびにコミット・プッシュを行う手間と時間の無駄。
  * 「コミット→プッシュ→CI失敗→修正」という長いフィードバックループ。
  * `Makefile`等のローカルタスクランナーとCI設定の二重管理。
* **想定利用者**:
  * GitHub Actionsを利用してCI/CDパイプラインを構築するすべての開発者。
  * CIのデバッグに時間を取られているエンジニア。
* **利用シーン**:
  * **CIワークフローのローカルデバッグ**: シンタックスエラーやスクリプトの動作をプッシュ前に手元で確認。
  * **ローカルタスクランナー**: `Makefile`の代わりにGitHub Actionsの定義を使ってビルドやテストを実行。
  * **クリーンな環境でのテスト**: 依存関係が含まれたDockerコンテナ上でテストを実行し、「自分のマシンでは動く」問題を排除。

## **3. 主要機能**

* **ワークフローのローカル実行**: `.github/workflows/` 内のYAMLファイルを解析し、Docker APIを通じてジョブを実行。
* **イベントトリガーのシミュレーション**: `push` や `pull_request` などのGitHubイベントをローカルで発火させてワークフローを実行。
* **アーティファクトの利用**: アクション間でのデータの受け渡しやアーティファクトの保存・利用をサポート。
* **シークレット管理**: ローカルの `.secrets` ファイルを通じて、APIキーなどの機密情報を安全に注入可能。
* **プラットフォーム指定**: `-P` オプションにより、ジョブを実行するDockerイメージ（Ubuntuなど）をカスタマイズ可能。
* **ドライラン**: 実際の実行を行わずに、どのジョブが実行されるかを確認する機能（`act -n`）。
* **特定のジョブのみ実行**: ワークフロー全体ではなく、特定のジョブを指定して実行可能（`act -j job_name`）。

## **4. 特徴・強み (Pros)**

* **高速なフィードバックループ**: コード変更からテスト結果確認までの時間を数分〜数十分から数秒〜数分に短縮できる。コミット履歴を汚さずに試行錯誤が可能。
* **GitHub Actionsとの互換性**: 標準のYAML構文をそのまま解釈するため、特別な設定ファイルを追加で書く必要がない。
* **環境の一貫性**: Dockerコンテナ上で実行されるため、ローカル環境の汚れや依存関係の差異に影響されにくい。
* **活発な開発**: 多くのコントリビューターにより頻繁にメンテナンスされており、新しいGitHub Actionsの機能への追従が早い。
* **クロスプラットフォーム**: Windows, Mac, Linuxのいずれでも動作（Dockerが必要）。

## **5. 弱み・注意点 (Cons)**

* **完全な互換性ではない**: GitHubホステッドランナーの環境を完全に再現できるわけではない。特にプリインストールされているソフトウェアの違いや、一部のGitHub固有のコンテキスト変数（`github.token`など）の挙動に差異がある場合がある。
* **Docker依存**: 実行にはDocker Engineが必要であり、Docker Desktop等のライセンスやリソース消費（CPU/メモリ）の影響を受ける。特にmacOSではファイルシステムのマウントが遅い場合がある。
* **「Docker in Docker」の複雑さ**: ワークフロー内でさらにDockerを使用する場合（Serviceコンテナなど）、特権モードの設定やパス解決などのトラブルが発生しやすい。
* **キャッシュ機能の制限**: GitHub Actionsのキャッシュ機能（`actions/cache`）は、ローカル環境では期待通りに動作しない場合や設定が必要な場合がある。

## **7. 導入実績・事例**

* **導入企業**: 特定の企業導入リストは公開されていないが、GitHub上のスター数は50,000を超えており（2026年1月時点）、世界中の多くの開発者やOSSプロジェクトで標準的なローカルテストツールとして利用されている。
* **導入事例**:
  * **OSSプロジェクト**: CIの設定変更時のレビュー負荷軽減のために、コントリビューターに`act`での事前確認を推奨するケース。
  * **個人開発**: プライベートリポジトリでのActions実行回数制限（無料枠）を節約するために、開発中のテストをローカルで行う。
* **対象業界**: Web開発、システム開発全般。

## **9. 連携機能 (API・インテグレーション)**

* **API**: ツール自体はCLIでありAPIは提供していないが、Go言語のライブラリとして他のツールに組み込むことは可能。
* **外部サービス連携**:
  * **Docker**: 実行環境として必須。
  * **GitHub**: GitHub APIと連携（必要に応じてトークンを使用）し、リポジトリの情報を取得可能。
  * **VS Code**: 公式拡張機能ではないが、サードパーティ製の拡張機能（GitHub Local Actionsなど）を通じてVS Code上から実行可能。

## **10. セキュリティとコンプライアンス**

* **認証**: GitHub Personal Access Token (PAT) を使用して、プライベートリポジトリのアクションやパッケージにアクセス可能。
* **データ管理**:
  * すべての処理はローカルマシン（およびローカルのDockerコンテナ）内で完結するため、ソースコードや認証情報が外部サーバーに送信されることはない。
  * `.secrets` ファイルによる環境変数の注入をサポート。**注意**: このファイルをGitにコミットしないよう `.gitignore` に追加することが強く推奨されている。
* **準拠規格**: OSSであるため特定のセキュリティ認証（SOC2等）の対象外だが、コードは公開されており監査可能。

## **11. 操作性 (UI/UX) と学習コスト**

* **UI/UX**: コマンドラインインターフェース（CLI）ベース。シンプルなコマンド体系（`act`, `act -l`, `act -j job`）で直感的に操作可能。実行ログはターミナルにカラー表示され、GitHub上のログと似た形式で確認できる。
* **学習コスト**: Dockerの基本的な知識があれば導入は容易。GitHub Actionsの知識が既にあるユーザーであれば、追加の学習コストはほぼ不要。`homebrew` や `choco` 等のパッケージマネージャーで簡単にインストール可能。

## **14. 類似ツールとの比較**

| ツール名 | 特徴 | 強み | 弱み | 選択肢となるケース |
|---------|------|------|------|------------------|
| **act** | GitHub Actions互換のローカルランナー | 既存のActions定義をそのまま使える。無料。 | 完全な再現性はない。Docker必須。 | GitHub Actionsを利用しており、ローカルで高速にテストしたい場合。 |
| **Dagger** | プログラマブルなCI/CDエンジン | 特定のCIサービスに依存せず、Go/Python/TSでパイプラインを記述。どこでも動く。 | 独自の記述方式を覚える必要がある。GitHub Actionsの資産は直接使えない。 | CI/CDのポータビリティを重視し、ベンダーロックインを避けたい場合。 |
| **CircleCI CLI** | CircleCIのローカルランナー | CircleCIの設定ファイルをローカルで実行・検証可能。 | CircleCIユーザー専用。 | CircleCIを利用している場合。 |
| **GitHub Actions (Cloud)** | 本家クラウド実行環境 | 完全な互換性。セットアップ不要。並列実行。 | フィードバックが遅い。デバッグが困難（SSHデバッグは可能だが手間）。 | 最終的な動作確認、本番デプロイ。 |

## **15. 総評**

* **総合的な評価**:
  GitHub Actionsを中心とした開発フローにおいて、`act` は**生産性を劇的に向上させるための必須ツール**である。クラウド上のCI環境とローカル環境のギャップを埋め、開発者の「待ち時間」を削減する効果は非常に大きい。完全な互換性がない点には注意が必要だが、それを補って余りあるメリットがある。
* **推奨されるチームやプロジェクト**:
  * GitHub ActionsをCI/CDに採用しているすべてのプロジェクト。
  * CIの設定変更が頻繁に発生するフェーズにあるプロジェクト。
  * ローカルでのビルド・テスト手順を標準化したいチーム。
* **選択時のポイント**:
  * 基本的には導入して損はない。ただし、「`act`で通ったから本番も通る」と過信せず、最終確認は必ずGitHub上で行うという運用ルールとセットで導入すべきである。
  * 複雑な環境構築が必要なジョブの場合、`act`用のカスタムDockerイメージを用意するコストとメリットを天秤にかける必要がある。
